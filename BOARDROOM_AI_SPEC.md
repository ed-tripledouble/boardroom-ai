# Board AI — Спецификация проекта (Project Specification)

## 1. Что это

Board AI — веб-приложение, которое симулирует совет директоров, экспертные команды и мультимодельные дискуссии из AI-моделей. Пользователь задаёт бизнес-вопрос, и 11 «директоров» (каждый на своей языковой модели с уникальным промптом) по очереди высказываются, после чего формируется итоговый протокол на русском языке.

Помимо совета директоров, есть 9 специализированных экспертных команд для глубокого погружения в конкретные области, а также режим «Дискуссия» — 3 универсальных эксперта на разных моделях обсуждают любой вопрос с последующим синтезом.

**Владелец:** Эд, основатель ДЭКАР Строй (аренда строительной техники, Москва).
**Основные проекты:** ДЭКАР Строй, агрегатор аренды техники, агрегатор запчастей для спецтехники.
**Базовая локация:** Россия (все рекомендации должны учитывать российское законодательство, санкции, доступность сервисов).

## 2. Архитектура

### Текущая версия (v4, браузерная)
- Один HTML-файл (`boardroom_ai_v4.html`), работает локально в браузере
- Название: **Board AI**
- API-вызовы напрямую к OpenRouter из браузера
- Хранение: localStorage (проекты, чаты, API-ключ)
- Загрузка файлов: Excel (.xlsx), PDF, Word (.docx), текст — парсинг через CDN-библиотеки (SheetJS, PDF.js, mammoth.js)
- Выборочное обращение к участникам: клик по чипу для выбора конкретных экспертов/директоров
- Сворачиваемый сайдбар (с инициалами в свёрнутом виде)
- Авторизация: **нет встроенной** — при деплое используется nginx basic auth
- Ограничения: привязка к одному браузеру/устройству (localStorage)

### Целевая версия (серверная)
- Фронтенд: текущий HTML или React/Next.js
- Бэкенд: Node.js (API прокси, хранение, авторизация)
- БД: PostgreSQL (проекты, чаты, контексты — синхронизация между устройствами)
- Хранение файлов: S3-совместимое (для Excel, PDF от пользователя)
- Авторизация: nginx basic auth (закрывает доступ до загрузки HTML)
- API-ключ: на сервере в `.env`, браузер не видит
- Деплой: VPS Selectel
- Доступ: с любого устройства через URL, мобильная адаптация

## 3. API

Все модели вызываются через **OpenRouter** (https://openrouter.ai/api/v1/chat/completions).
Один API-ключ, единый формат запросов.

### Используемые модели:
| Модель | Идентификатор OpenRouter | Для чего |
|--------|--------------------------|----------|
| Claude Sonnet | anthropic/claude-sonnet-4 | Аналитика, логика, стратегия |
| GPT-4o | openai/gpt-4o | Креатив, идеи, широкий кругозор |
| Perplexity Sonar Pro | perplexity/sonar-pro | Поиск свежих данных из интернета |

### Формат вызова:
```
POST https://openrouter.ai/api/v1/chat/completions
Headers: Authorization: Bearer <key>, Content-Type: application/json
Body: {
  model: "<model_id>",
  messages: [
    { role: "system", content: "<GLOBAL_RULES + персональный промпт>" },
    { role: "user", content: "<контекст предыдущих ответов + вопрос + бизнес-контекст>" }
  ],
  max_tokens: 1500,
  temperature: 0.7
}
```

## 4. Структура команд

### 4.1. Совет директоров (11 человек + протокол)

Совет — основной уровень. Стратегические решения, оценка новых направлений, ключевые бизнес-вопросы.

| # | Роль | Модель | Специализация |
|---|------|--------|---------------|
| 1 | CEO / Стратег | Claude Sonnet | Видение, приоритеты, решения |
| 2 | Финансовый директор | Claude Sonnet | ROI, риски, бюджеты, юнит-экономика |
| 3 | Коммерческий директор | GPT-4o | Продажи, рынок, конкуренты |
| 4 | Операционный директор | Perplexity | Логистика, процессы, масштабирование |
| 5 | Директор по маркетингу | GPT-4o | Каналы, бренд, привлечение |
| 6 | Юрист | Perplexity | Законы РФ, лицензии, регуляторика |
| 7 | IT-директор | Claude Sonnet | Технологии, стеки, архитектура |
| 8 | HR-директор | GPT-4o | Команда, найм, оргструктура |
| 9 | Директор по развитию | Claude Sonnet | Новые продукты, ниши, MVP, PMF |
| 10 | Директор по инновациям | Claude Sonnet | Дизрапт, прорывные идеи, международный опыт |
| 11 | Адвокат дьявола | Claude Sonnet | Слабые места, жёсткая критика |
| 12 | Протокол (синтезатор) | Claude Sonnet | Итоговая сводка на русском |

**Порядок выступлений:** строго по номерам. Адвокат дьявола всегда последний перед протоколом.

**Каждый директор видит ответы всех предыдущих** — это критично для качества, чтобы не повторялись и дополняли друг друга.

### 4.2. Экспертные команды (9 команд)

Второй уровень — глубокое погружение в конкретную область. Запускаются двумя способами:
1. **Автоматически:** директор на совете запрашивает экспертную команду через маркер `[DRILL:team_id:вопрос]`
2. **Вручную:** пользователь заходит в команду напрямую через сайдбар

| Команда | Количество экспертов |
|---------|---------------------|
| Маркетинг | 7 |
| Коммерция / Продажи | 7 |
| Финансы | 5 |
| IT / Технологии | 6 |
| Юридический | 5 |
| HR / Оргразвитие | 5 |
| Операции / Логистика | 5 |
| Стратегия / Аналитика | 5 |
| Продукт / Развитие | 5 |

**Итого:** 11 директоров + 50 экспертов + 1 синтезатор = **62 AI-роли**

## 5. Механика работы

### 5.1. Заседание совета директоров
1. Пользователь задаёт вопрос
2. Каждый директор получает: GLOBAL_RULES + свой промпт + бизнес-контекст проекта + **история чата (последние 20 сообщений)** + ответы предыдущих директоров в текущем раунде + вопрос
3. Директора отвечают на английском (для качества моделей)
4. Если директор считает нужным — добавляет маркер `[DRILL:team_id:вопрос]`
5. Система перехватывает маркер, запускает экспертную команду, показывает результат
6. После всех директоров (если 2+) — синтезатор собирает протокол на русском

**История чата:** директора помнят предыдущие вопросы и ответы в рамках текущего чата. Ответы в истории обрезаются до 300 символов для экономии токенов. Прикреплённые файлы также попадают в историю.

### 5.2. Авто-углубление (drill)
- Директор добавляет в конец ответа: `[DRILL:marketing:Какие каналы привлечения оптимальны для B2B аренды?]`
- Система запускает маркетинг-команду с этим вопросом
- Все 6 экспертов отвечают по очереди
- Результат показывается в чате как вложенный блок
- Результат также передаётся следующим директорам как контекст

### 5.3. Прямое обращение к команде
- Пользователь выбирает команду в сайдбаре
- Задаёт вопрос напрямую
- Все эксперты команды отвечают по очереди
- Протокол не формируется (только для совета директоров)

### 5.3.1. Выборочное обращение к участникам

Пользователь может задать вопрос не всей команде/совету, а конкретным участникам:

**Механика:**
- Клик по чипу участника в полоске → чип выделяется (синяя рамка + фон)
- Клик ещё раз → снимается выделение
- Можно выбрать одного или нескольких участников
- Если есть выбранные → вопрос уходит только им
- Если никто не выбран → вопрос уходит всем (стандартное поведение)
- Кнопка "сбросить" для быстрой очистки выбора
- При переключении команды/проекта → выбор сбрасывается автоматически

**Протокол при выборочном режиме:**
- Если ответило 2+ участника — протокол формируется
- Если ответил только 1 участник — протокол не нужен

**Примеры использования:**
- Задать вопрос только CEO и финансовому директору
- Спросить только SEO-специалиста в команде маркетинга
- Быстро уточнить у юриста один вопрос, не гоняя весь совет

**Вся история остаётся в контексте:** выборочные ответы сохраняются в общий чат команды наравне с полными заседаниями.

### 5.4. Ознакомительная сессия (onboarding)

**Обязательный режим при первом запуске проекта.** Перед первым полноценным заседанием система проводит ознакомительную сессию:

1. Пользователь загружает контекст проекта (текст + файлы)
2. Система запускает режим "Знакомство": каждый директор по очереди читает контекст и задаёт уточняющие вопросы
3. Порядок: CEO первый, далее по номерам. Каждый видит вопросы предыдущих (чтобы не дублировать)
4. Директор может спросить: недостающие цифры, уточнения по бизнес-модели, финансовые данные, юридический статус и т.д.
5. Пользователь отвечает (текстом или файлами)
6. Процесс повторяется до тех пор, пока директора не скажут "достаточно информации"
7. После этого контекст считается заполненным, и можно начинать полноценные заседания

**Зачем:** без фундаментальных данных совет будет гонять пустоту. Лучше потратить 20 минут на вопросы, чем получить абстрактные советы.

**Технически:** отдельная кнопка «🎓 Знакомство» в интерфейсе. Промпт для этого режима:
```
"This is an ONBOARDING SESSION — not a regular meeting. You are reviewing the project context for the first time. 
DO NOT give advice yet. Instead:
1. Read the provided context carefully
2. Identify what critical information is MISSING for you to do your job well
3. Ask 2-5 specific, concrete questions about what you need
4. Explain briefly WHY you need each piece of data
5. If you have enough info for your area — say so explicitly
Format your questions as a numbered list. Be specific: not 'tell me about finances' but 'what is the monthly revenue and net margin for the last 6 months?'"
```

### 5.5. Загрузка файлов в контекст

**Два типа данных:**

| Тип | Куда загружать | Примеры | Когда обновлять |
|-----|---------------|---------|-----------------|
| Постоянные | Контекст проекта (кнопка 📝) | Реквизиты, сайт, структура компании, парк техники, прайсы, оргструктура | Один раз, редко обновлять |
| Оперативные | В чат перед вопросом | Отчёт за месяц, воронка продаж, финансы за квартал, свежие метрики | Каждый раз перед заседанием |

**Поддерживаемые форматы (текущая версия, браузерная):**
- Excel (.xlsx, .csv) — парсится в текстовую таблицу
- PDF — извлекается текст
- Текстовые файлы (.txt, .md)
- Изображения — пока не поддерживаются (планируется)

**Механика (браузерная версия):**
- Кнопка «📎 Файл» рядом с полем ввода
- Файл парсится в текст на клиенте (через SheetJS для Excel, PDF.js для PDF)
- Текст добавляется к сообщению пользователя как вложение
- Директора видят содержимое файла в контексте своего запроса

**Механика (серверная версия, будущее):**
- Раздел «Файлы проекта» в сайдбаре
- Загрузка в облачное хранилище
- Автоматический парсинг и индексация
- RAG: релевантные фрагменты подтягиваются к каждому запросу

### 5.6. Дискуссия (мультимодельный режим)

Режим для глубокого обсуждения любого вопроса тремя независимыми AI-экспертами на разных языковых моделях. Доступен внутри каждого проекта — эксперты получают контекст проекта.

**Участники (всегда все трое, выбор отдельных отключён):**

| # | Роль | Модель | Стиль мышления |
|---|------|--------|----------------|
| 1 | Аналитик | Claude Sonnet | Системный анализ, логика, структура, риски |
| 2 | Практик | GPT-4o | Реализация, подводные камни, альтернативы, опыт |
| 3 | Исследователь | Perplexity Sonar Pro | Свежие данные, мировые аналоги, тренды, факты |
| 4 | Синтезатор (⚡ Итог) | Claude Sonnet | Консенсус, разногласия, лучший ответ, следующие шаги |

**Механика:**
1. Пользователь задаёт вопрос (любой: код, бизнес, стратегия, логика)
2. Аналитик отвечает первым — даёт структурированный анализ
3. Практик отвечает вторым — видит ответ Аналитика, комментирует, добавляет практический взгляд
4. Исследователь отвечает третьим — видит обоих, подкрепляет или опровергает данными
5. Синтезатор подводит итог: консенсус, разногласия, лучший ответ, что нужно дополнительно

**Ключевые отличия от экспертных команд:**
- Универсальные роли — не привязаны к конкретной области
- Ценность в разнице моделей, а не ролей
- Всегда все 3 + итог, без выбора отдельных
- Подходит для ревью кода, обсуждения идей, проверки гипотез

**Контекст:** история чата (последние 20 сообщений) + контекст проекта передаются в каждый запрос. Несколько вопросов подряд — эксперты помнят предыдущие обсуждения.

**Бэклог (серверная версия):**
- Отдельные треды дискуссий внутри проекта (сейчас один чат на проект)
- Режим «Мозговой штурм»: дебаты с раундами возражений
- Принудительный drill-down: директор опрашивает свою команду по указанию пользователя

## 6. Общая инструкция (GLOBAL_RULES)

Получают ВСЕ 62 роли с каждым запросом. 16 правил:

1. **Сначала пойми задачу.** Если неясно — переспроси. Лучше переспросить, чем дать поверхностный ответ.
2. **Жёсткая честность.** Плохая идея = "это плохая идея" + почему. Никакого "интересный подход, но...".
3. **Не знаешь — скажи.** Никогда не выдумывай. Указывай уровень уверенности: "уверенность ~60%, это гипотеза".
4. **Конкретика.** Цифры, сроки, названия инструментов, цены. Не "нужно продвигаться" — а "Яндекс.Директ, бюджет от 50К/мес, CPL 800-1200 руб".
5. **Предлагай альтернативу.** Не останавливайся на первом варианте. "Вариант А, но лучше рассмотреть Вариант Б, потому что..."
6. **Инструменты и ограничения.** Нужен внешний сервис/API — скажи прямо: "Для точного ответа нужно подключить [сервис], вот что он даст".
7. **Международные аналоги — обязательно.** США, Европа, Китай, Корея, Индия. Если нигде нет — придумай сам. Комбинируй идеи из разных индустрий.
8. **Российская реальность + обходные пути.** Если сервис заблокирован — дай ОБА: лучший глобальный (название, что делает, заблокирован, нужен VPN + зарубежная карта) + доступную российскую альтернативу. Пусть пользователь решит.
9. **Не повторяй других.** Ссылайся на их мнения и фокусируйся на упущенном.
10. **Формат:** 200-400 слов. Без вступлений, без "в заключение". Нумеруй пункты, без воды.
11. **Ошибочная предпосылка** — если вопрос задан неправильно или идея ошибочна, скажи первым делом.
12. **Запрашивай реальные данные.** Нужны цифры бизнеса — попроси конкретно: "Мне нужен отчёт по марже за 6 мес в Excel". Без данных пиши: "Мой ответ — гипотеза, для точного анализа пришлите [что]."
13. **Релевантность.** Оцени, применима ли твоя специализация к данному бизнесу. Если нерелевантна — скажи одним предложением почему и предложи, что подойдёт вместо. Не трать время на развёрнутый ответ если твой канал/подход здесь не работает.
14. **Примеры — ориентир, не окончательный список.** Когда называешь сервисы, банки, компании — всегда ищи лучшие варианты. Сравнивай минимум 3-4 альтернативы.
15. **AI-угроза.** При оценке любого бизнеса/продукта — оценивай риск замещения AI. Могут ли клиенты решить задачу сами с помощью AI? Через какой горизонт? Что менять в продукте?
16. **AI-first мышление.** При любой рекомендации — оценивай, можно ли задачу решить лучше/дешевле/быстрее с помощью AI. AI-решение предлагай первым, традиционный подход — как fallback. Называй конкретные инструменты и API.

## 7. Персональные промпты директоров

## 12. Перспективные фичи (бэклог)

### 12.1. Принудительный опрос команды директором

Когда выбран конкретный директор из совета — кнопка/тогл «Опросить команду». Механика:
1. Система определяет, какая экспертная команда относится к этому директору (CMO → маркетинг, CFO → финансы, CTO → IT и т.д.)
2. Сначала опрашивается вся экспертная команда по вопросу пользователя
3. Затем директор получает вопрос + ответы своих экспертов → даёт итоговый ответ с учётом их мнений
4. Всё сохраняется в общий чат проекта

Отличие от текущего drill: сейчас директор сам решает, нужно ли углубление. Новая фича — пользователь принудительно запускает консультацию.

### 12.2. Мозговой штурм (мультимодельный фильтр)

Отдельная команда «Мозговой штурм» с двумя режимами:

**Режим «Мультимодель»** — одна роль, разные модели:
- 3 одинаковых эксперта на разных моделях (Claude, GPT-4o, Perplexity)
- Каждый видит ответы предыдущих и может возразить
- Синтезатор собирает лучшее
- Применение: код-ревью, проверка фактов, аудит документов

**Режим «Мультиперспектива»** — разные роли, разные модели:
- Аналитик (Claude) — логика, структура, риски
- Креативщик (GPT-4o) — идеи, нестандартные подходы
- Исследователь (Perplexity) — свежие данные, аналоги, факты
- Синтезатор (Claude) — объединяет лучшее, отмечает согласие/споры, финальная рекомендация
- Применение: бизнес-идеи, стратегия, выбор ниши, обсуждение рынка

Пользователь выбирает режим при запуске вопроса.
